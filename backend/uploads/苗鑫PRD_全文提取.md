2. 业务背景及目标（必须有）

2.1 业务背景

产品与公司：本产品为「AiPPT.com的Agent模式」；核心能力为：用户用自然语言提需求，Agent 自动生成符合 Marp（或 Reveal.js）规范的幻灯片源码（Markdown/HTML/CSS），再经渲染交付给用户。

C 端用户典型场景：课程答辩、职场汇报、活动分享等——共性是「想快、想好看、想少动脑」，且多数用户不具备 Marp/Reveal 书写能力。

C 端典型痛点

从零搭结构累：大纲、分页、讲稿备注耗时间。

模板与审美不稳定：网上模板杂、自己调版心累。

“瞎编”与不适配：纯大模型生成易事实漂移、与用户上传资料不一致；需 RAG（用户资料 + 平台可控知识） 与 后处理（内容安全/敏感词） 兜底（课程思路：检索增强 + 工程规则）。

交付形态：C 端更关心 在线预览、一键导出 PPTX/PDF、分享链接，源码能力可作为「进阶/导出附件」而非主路径门槛。

2.2 产品目标（业务目标 + 产品目标）

业务目标（C端）

增长与留存：首屏到「可用一版」路径极短；生成成功率与满意度驱动次日/7日留存。

商业化：会员/次数包/导出权益清晰；给用户免费额度试用，提升付费转化。

产品目标

输入：一句话/多轮对话需求 +（可选）参考资料（文件/链接）+ 轻量表单（页数、风格、场景）。

输出（对内可复用、对外可交付）：

用户主路径：可预览的幻灯片 + 导出 PPTX/PDF（优先）。

技术真源：Marp slides.md + theme.css + assets/ + manifest.json，或 Reveal index.html + CSS + assets/（可用于进阶导出、调试与迭代）。

生成链路工程化：意图识别→路由→RAG检索→结构化规划（JSON）→逐页生成→后处理→渲染预览→导出。

2.3 关键成功指标（北极星，C端）

一次生成可用率：用户认为「基本能拿去讲/交」的比例 ≥ 70%（V1），≥ 85%（V2/V3）（问卷/点踩/重生成率综合）。

平均生成时延：20页以内，95%的用户生成时间 ≤ 180s（V1）。

内容相关：涉政涉暴涉黄等命中拦截率、误杀率98%；用户上传资料场景的 引用可追溯（有资料时）≥ 90%（V2起，作为质量项而非全员感知指标）。

模板美观一致性：选定风格下「版式不翻车」，（无溢出/重叠）占比 ≥ 95%（渲染质检）。

3.3 分析维度

以下维度与 C 端 AiPPT 当前阶段 对齐：

用户路径与心智：从进入到「可导出」的步骤数、是否需要理解专业概念（Markdown/HTML）。

输入形态：纯主题 / 多轮对话 / 上传文档与链接 / 混合编辑。

生成与可控性：大纲是否可改、单页重生成、风格与模板数量、是否强调引用与溯源。

输出形态：PPTX、PDF、HTML、是否可编辑、是否支持「源码级」复用与版本管理。

差异化技术资产：是否沉淀 Marp/Reveal 标准输出（便于校验、主题复用、A/B、成本优化）。

生态与护城河：模型与工具集成广度 vs 垂直场景打磨深度（模板、质检、内容安全、失败兜底）。

商业化与成本：免费额度、会员、算力与缓存策略对毛利的影响。

3.9 建议与总结

3.9.1 竞争策略

对外一句话：少讲「Marp/Reveal」，多讲 「按你的资料/主题，几分钟导出能讲的 PPTX，版式少翻车」；技术栈作为 「我们更稳可迭代」 的 B 端/投资人叙事备用。

差异化切口（三选一或组合）：

资料驱动：上传 PDF/Word 后强对齐要点 + 引用展示（高级面板折叠）；

场景模板：答辩 / 转正 / 竞品分析 / 读书分享等 成套结构；

导出与质检：同题对比竞品，把 Office 打开效果、溢出/糊图率 做成内部北极星。

对 Genspark：避免拼「全能」；拼 中文场景、本地支付与合规、幻灯片垂直链路时延与成功率。

对天工：避免只拼「也能生成 PPT」；拼 细分人群、审美、单页重生成、会员权益清晰度、失败兜底话术。

3.9.2 版本优先级

| 优先级 | 动作 |
| --- | --- |
| P0 | 主路径：生成 → 预览 → 导出 PPTX/PDF 成功率与耗时；移植AiPPT模版站的优质模版。单页重生成；点踩与 bad case 埋点。 |
| P1 | 上传资料 + RAG； |
| P2 | 分享链接、会员权益、缓存与配额防刷。 |
| P3 | 进阶导出 Marp/Reveal 源码（极客与口碑传播）。 |

3.9.3 总结

Genspark 强在 Agent + 多模型 + 一体化工作流，幻灯片是其能力之一。

天工 强在 国内办公心智、中文 PPT 场景与渠道。

AiPPT——网页生成PPT模式，需把 「幻灯片垂直链路 + 导出体验 + 场景模板 + 资料对齐」 做到用户可感知，Marp/Reveal 作为 工程质量与长期壁垒。

4. 产品方案

4.1  产品架构

逐步生成：

1、AI生成大纲（依据是用户query+联网搜索，最复杂的情况，支持用户上传文档）——>支持人工编辑

2、AI生成每页的内容（结合平台的RAG知识库+联网搜索）——>支持人工编辑

3、每页具体文案的AI润色/改写（LLM）

4.1.1 角色与使用场景（C端）

普通用户（核心）：学生、职场新人、博主、教师等；输入需求（+可选资料），要「快、好看、能导出」。

会员用户：更高页数上限、更多模板/风格、优先队列、高清导出、去水印等（具体权益产品定）。

游客/新用户：极低门槛试用（有限次数/有限页数/带品牌水印）。

平台运营/产品（后台）：配置模板主题、场景骨架、敏感词与免责声明策略、活动运营位；非企业客户的 IT 管理员角色。

客服/风控：处理投诉、误杀申诉、违规内容处置（轻量流程即可）。

4.1.2  典型流程（用户视角，C端主路径）

打开 AiPPT → 选场景/风格（可选）→ 输入一句话需求（+上传资料可选）→ 生成预览 → 微调文案/换图/单页重生成 → 导出 PPTX 或 PDF → （可选）分享链接 / 保存到我的作品。

进阶路径：查看/下载 Marp 或 Reveal 源码包（面向极客或合作方，可放在「更多导出」）。

5.1.5  需求理解与预处理（检索前优化）

这一段在系统里对应「检索前优化」：先把用户话说明白，再决定去哪个知识库、用什么模板、检索什么关键词。否则 RAG 会把噪声当信号，后面越生成越偏。

Query 改写（必须）输入：多轮 conversation_history + 当前 user_query。输出：一份结构化任务描述（建议 JSON 或固定段落模板），至少包含：goal（目的）、audience、slide_count_hint、must_cover[]、must_avoid[]、style_keywords[]、time_scope（若涉及数据/新闻）、domain（领域标签）。改写失败（解析非 JSON、关键字段全空）时：重试 1 次；仍失败则用原始 query 直通意图模型，并在日志打 rewrite_fallback=true 供评测抓 bad case。

意图识别（必须）输出：intent（闭集 intent_id，与 §5.1.8.4 意图字典一致，版本与埋点 intent_taxonomy_version 对齐）、confidence（0–1 浮点）、candidates[]（Top3–5，含 intent_id 与 prob 或分数）。产品侧规则：三档阈值 仅 §5.1.8.2：≥0.85 自动；0.7～0.85「我猜你是」；<0.7 多轮意图澄清。易混意图的规则增强（如「有附件+根据」偏上传生成）见 §5.1.8.5，不得绕过三档（规则只可影响候选排序或二次打分，最终仍以 Top1 置信度落档为准）。服务端须注入 session_stage，防止在「新建页」误路由到 任务内编辑类意图（如 EDIT_PAGE_REGEN）。

槽位抽取（必须）输出槽位覆盖：industry、product_name、client_name、time_range、metric_definition（口径）、language、compliance_flags（是否涉投教/医疗免责声明等）。每个槽位带来源标记：user_explicit / inferred / unknown。只有 unknown 且命中「必填策略」时才进入澄清 UI；inferred 必须在结果页可一键修改（避免用户被静默代表）。

5.1.7  模板与主题（Marp/Reveal，C端）

模板在 C 端不只是「好看」，而是稳定性：同一套主题 CSS 决定了标题字号、行距、配色、代码块样式，能直接降低「生成对了但版式炸了」的概率。前台展示上，建议每个主题卡片带 缩略图 + 适用场景一句话 + 是否会员专享；用户点选前可先弹「三页样例预览」，减少选错后的整包重生成。

Marp 线：theme.css、字体声明（含 Web 字体授权信息）、封面/章节过渡页样式、页脚/页码、可选平台 Logo 与水印规则（免费用户是否强制水印由 5.4.10 与商业策略决定）。

Reveal 线：主题 CSS、reveal.js 初始化片段（过渡动画、键盘行为）、常用 layout 片段（双栏、时间线、卡片墙），保证 Agent 生成的 HTML class 与主题约定一致。

内容安全包：敏感词库分三级（见 5.4.9）、替换话术、按场景插入的免责声明（例如泛财经类「非投资建议」——具体文案法务审核）。

5.1.8  预览与编辑（源码友好）

预览页是用户判断「值不值得导出」的唯一依据，建议信息架构分三层，避免一次塞满吓跑小白：

默认层：大屏逐页预览 + 左侧缩略图 + 顶部导出按钮（主路径）。

编辑层：改某一页标题/要点/讲者备注；换图；对单页触发「重生成」（走后端单页接口，超时与重试见 5.4.7）。

高级层：折叠面板里提供 MD/HTML/CSS 源码与 引用来源（按 citations 高亮对应资料片段），满足较真用户与客诉复盘。

不满意反馈不要只放一个 thumbs down：用结构化原因（结构不对 / 口径与资料不符 / 风格不一致 / 合规担心 / 配图不合适 / 页数不对 / 其他），原因直接进入看板 TopN，驱动每周迭代优先级。

5.1.9  导出与分享

导出是转化与口碑的临门一脚。PPTX 建议 V1 就做到「WPS/Office 能打开、字体不疯狂报错、动画可关」——兼容性问题要在测试用例里写死常见版本。PDF 用于「只读分发」，应保留矢量文字与嵌入字体策略说明（产品可对用户一句话：「PDF 更适合发给别人，PPTX 更适合自己再改」）。

源码包导出（slides.md 或 index.html + theme.css + assets/ + manifest.json）适合进阶用户与二次开发合作方，可放在「更多导出」里，避免主路径分心。

分享链接：只读、可设密码与有效期；免费用户可强制水印或限制打开次数（与 5.4.10 一致）。每一次分享生成独立的 share_id，便于撤销。

版本管理：同一任务下保留每次生成的版本，对比维度至少包括：页数、总字数、引用覆盖率、渲染告警数、用户是否导出成功——方便用户「回到上一版」。

5.1.10  兜底策略（必须，生产可用）

兜底策略的目标是：任何一步挂了，用户仍能拿到可解释的结果或下一步，而不是无限转圈或空白页。

三层生成（与课程「工程优先 + 模型兜底」一致）1）模板骨架命中：意图明确、资料一般时，优先走骨架 + 轻量补全，成本低、结构稳。2）RAG + 规划 + 逐页生成：用户有资料、需要可追溯事实时的主路径。3）大模型长尾兜底：骨架与 RAG 都吃不下的需求，允许更自由生成，但必须附带更强的风险提示与后处理。

节点失败：改写、规划、单页生成、渲染、导出各自 独立重试与降级（次数/超时见 5.4.7）。整任务失败时，返回明确错误码（如 MODEL_TIMEOUT、RENDER_OOM、EXPORT_UNSUPPORTED_FONT），前端映射成用户能懂的一句话 + 「重试」按钮。

资源失败：图片拉取失败 → 占位图 + 「缺失素材列表」；字体缺失 → 回退字体族并记录 render_warnings，避免导出打开后全盘崩版。

5.1.11 意图体系

产品定义的闭集分类 + 可路由的承接动作。把每一类意图拆到「名称、描述、典型说法、层级、边界、承接动作、兜底」——并在工程侧与 §5.1.8.2 的置信度阈值、节点 C 路由、骨架模板显式挂钩。

5.1.11.1  设计原则

闭集 + 版本号：所有 intent_id 由产品维护字典表，研发与埋点只引用 ID，不现场造新枚举；变更走 intent_taxonomy_version（见 7.1）。

先判「会话阶段」再判意图：同一句话在「新建任务页」与「已有作品的预览页」含义不同——例如「把第二页改短」在新建页应视为 主题口述 的素材，在预览页则是 单页编辑。系统应用 session_stage：new_task | awaiting_clarify | editing_existing_task 做硬约束（非法组合直接降级为澄清）。

置信度统一真源：≥0.85 / 0.7～0.85 / <0.7 三档行为一律遵守  §5.1.8.2；

互斥与分层：L2 意图在同一 session_stage 下尽量 MECE（相互独立、共同穷尽）；确有交叉时用 优先级规则（见 5.1.8.5）或合并为上级 L1 再二次分类。

5.2  非功能需求

性能：20 页 Deck 端到端 P95 ≤ 180s（V1）；单页重生成 P95 ≤ 20s（V2 目标）。渲染与导出在工程上应 流水线并行（例如后一半页面渲染时前一半已开始导出准备），但产品对外仍展示「总进度条」以免用户误以为卡死。

稳定性：任务状态机建议细分为：created → understanding（含澄清等待）→ indexing（仅有附件时）→ retrieving → planning → awaiting_outline_confirm（可选）→ generating → rendering → post_processing → done / failed。失败任务支持从中间节点恢复（例如仅重跑渲染与导出），减少用户重复花钱跑全链路。

安全与合规（C端）：用户资料与向量索引 默认仅本人可检索；账号注销时 7 天内可撤销，期满级联删除对象与索引。输出必须经过 5.4.9 分级处置；未成年人模式（若上线）单独词库与拦截策略。日志留存周期符合隐私政策，且展示给用户的文案要写清「何种数据用于改进模型」（若公司采用用户数据训练，需单独同意，不在本 PRD 默认范围内）。

可观测：每个任务记录一条 trace id；关键节点保留 脱敏后的输入输出摘要（禁止存用户全文身份证、银行卡等）。告警阈值示例：某模板 render_warnings 率突增、某意图澄清率 >40%（说明分类或文案有问题）。

成本控制：缓存键与 TTL 见 5.4.8；大模型调用按 5.4.10 配额硬挡，防止黑产刷接口。可选：高峰期限速时优先保证会员队列（公平性策略产品需公示）。

6.1  业务背景概述

平台核心矛盾（AiPPT / C端）：

用户表达口语化、信息不全、且期望极短路径拿到「能讲的一版」，必须做 改写/意图识别/澄清（课程：检索前优化）。

纯生成易 幻觉 且易触发 内容安全风险；用户上传资料时还需 RAG 对齐资料。

因此采用“意图/路由 + RAG（用户资料+平台库）+ 强规则后处理 + 缓存 + 观测评测闭环”的生产级方案。

6.2  业务流程（端到端）

6.2.1  流程（高层）

需求输入 → 改写/意图/槽位补全 → 路由（数据源/工具/模板） → RAG检索 → 结构化规划JSON → 逐页生成源码 → 渲染预览 → 后处理（合规/脱敏/规范校验）→ 导出/分享

6.2.2  关键节点的输入输出定义（给研发/算法的接口约束）

节点A：Query改写

输入：conversation_history、user_query

输出：rewritten_query（结构化文本/JSON）、missing_slots[]

节点B：意图识别 + 置信度

输入：rewritten_query

输出：intent、confidence、candidates[]

节点C：路由

输入：intent、slots

输出：data_sources[]、tools[]、strategy、framework

节点D：RAG检索（混合检索+重排）

输入：rewritten_query、data_sources

输出：chunks[]（含来源与权限）、rerank_scores

节点E：Slide规划

输入：rewritten_query、chunks[]

输出：slides_plan(JSON)（强约束：只允许JSON）

节点F：逐页生成（Marp/Reveal代码）

输入：slides_plan、chunks[]、theme_constraints

输出：md/html/css + assets_manifest.json + citations_map

节点G：渲染预览

输出：page_images[]、render_warnings[]

节点H：后处理（强规则）

输出：risk_report、desensitize_report、fixed_files

6.3  交互与页面（MVP）

6.3.1  我的作品 / 任务列表

列表字段：封面缩略图、任务名、模板风格、状态、耗时、是否已导出、最近更新时间

操作：继续编辑、复制生成、删除、重试失败

6.3.2  创建任务页

需求区：自然语言 + 结构化参数（页数、受众、语气、语言、输出框架）

资料区：上传/选择资料（含权限提示）

模板区：Marp/Reveal模板卡片 + 预览

策略区：标准/快速/精细

6.3.3  生成结果页（预览/编辑/导出）

左侧：页缩略图 + 质量告警（溢出/低对比度/缺图/合规命中）

右侧：大屏预览为主；源码/引用来源放入「高级」折叠（降低 C 端认知负担）

顶部：导出 PPTX、PDF；次要入口：源码包、版本对比

反馈：不满意原因 + 点踩；驱动 bad case 迭代

6.4 生成策略（生产级：分层与兜底）

6.4.1  规则优先

Deck骨架模板库（按意图命中）

结构规则：每页信息密度上限、标题长度上限、要点条数上限

合规规则：免责声明强制插入、敏感词替换

6.4.2  模型辅助（只做可控部分）

摘要、标题口吻、版式建议（layout_hint）

风险控制：必须带 citations；缺失/冲突则回退模板骨架并提示补充资料

6.4.3  后处理（强规则）

敏感词检测 + 替换话术模板（课程：不能依赖大模型100%遵循）

脱敏：手机号/证件号/地址/银行卡号掩码规则

规范校验：Marp/Reveal结构、CSS合法性、资源引用可达

渲染后质量校验：溢出/字号过小/对比度不足

6.5  权限与数据隔离（C端必备）

用户级隔离：A 用户不可见 B 用户作品与资料

会员与功能权益：导出清晰度、页数上限、去水印、优先队列等与订阅绑定

分享：仅持有人默认可编辑；访客只读和下载

6.6  评测与迭代（0到N）

6.6.1  离线评测集（必须建设）

构建“需求→Deck”的评测集：覆盖不同意图与资料组合

评测指标（示例）

内容覆盖/完整性：必须出现要点是否缺失

引用覆盖：关键事实是否有来源

渲染质量：溢出/对比度/缺图

模板/风格一致性：主题命中率（视觉与版式）

6.6.2  线上bad case闭环（必须）

典型bad case信号（课程写法）

用户连续相似需求重复生成

用户点“不满意”且原因集中

导出前停留时间过长（大量修改）

闭环：每周Top问题→规则/模板/检索/提示词迭代→离线评测对比

7. 数据及用户行为采集需求（必须有，要给开发提需求）

P95时延

9. 商业模式

会员订阅：月度/年度；含页数、导出次数、模板、去水印、优先队列等权益组合。

ToB / 私有化：非本期核心，可作为远期产品线。

附录A：术语表

意图识别（Intent）：将用户需求映射为可执行的业务分类（课程：分类+置信度）

路由（Routing）：根据意图/槽位决定数据源与工具调用路径（课程：where + how）

RAG：检索增强生成；C端场景主要为 用户上传资料 + 平台可控公共知识，降低幻觉与资料不一致

slides_plan：强约束结构化规划JSON，驱动逐页生成

后处理：强规则合规/脱敏/规范校验/自动修复

bad case：线上表现不佳样本（不满意/重生成/导出前大改），用于迭代闭环
